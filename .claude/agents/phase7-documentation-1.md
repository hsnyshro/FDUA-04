# Phase 7：事業計画書初稿構成段階

## 入力データ
output/[企業コード番号]/ 配下のファイル一覧

---
## 🎯 使命

Phase1-6の全分析結果と戦略提案を統合し、事業計画書の初稿を12要素フレームワークに基づいて構成します。

**対応する12要素**: ①～⑫ すべての要素を統合した初稿作成


## 📋 実施内容

### 1. 12要素の統合構成

事業計画書を以下の12要素で構成：

| # | 要素 | 内容 | 出典 |
|----|------|------|------|
| ① | ビジョン・目標 | 企業の経営理念、定量的目標 | Phase1, 4 |
| ② | ビジネスコンセプト | Who, What, How | Phase1, 3, 5 |
| ③ | ターゲット市場 | 顧客セグメント、市場規模 | Phase2, 3 |
| ④ | 市場環境 | PEST分析、業界トレンド | Phase2 |
| ⑤ | 競合分析 | 競合ポジショニング | Phase2, 3, 5 |
| ⑥ | ビジネスモデル | 価値提案、収益モデル | Phase3, 5, 6 |
| ⑦ | マーケティング戦略 | 4P戦略、営業施策 | Phase6 |
| ⑧ | 損益計画 | 売上予測、利益計画 | Phase6 |
| ⑨ | リスク評価 | リスク要因と対応策 | Phase5 |
| ⑩ | 実装計画 | ロードマップ、マイルストーン | Phase6 |
| ⑪ | 財務予測 | 3～5年の財務見通し | Phase6 |
| ⑫ | サマリー・結論 | 提案の概要と期待効果 | 本フェーズ |

### 2. 初稿の構成

```
# [企業名] 経営戦略事業計画書

## エグゼクティブサマリー
（⑫ サマリー）

## 1. 現状分析
（①②③④⑤⑥ の統合）
- 企業概況
- 市場・競合環境
- 内部資源・SWOT
- 経営課題

## 2. 戦略提案
（⑥⑦⑧⑨⑩⑪ の統合）
- ビジネスモデル
- マーケティング戦略
- 実装計画
- 財務計画

## 3. 期待効果
（⑫ 結論）
- 定量効果
- 定性効果
- リスクと対応

## 付録
- 詳細分析資料
- 参考資料
```

### 3. 文字数・品質管理

- **目安**: 約10,000字程度
- **構成**: 要素①～⑫がすべて網羅されているか確認
- **整合性**: Phase1-6の分析と矛盾がないか確認
- **表現**: 論理性、説得力、読みやすさ

## 🔧 実装指示

以下のプロセスに従ってファイルを生成してください：

1. **全前フェーズのファイルを読み込み** - Phase1-6に加えて、past-future.md、regional-analysis.md、construction-industry-channel.md、gx-dx-strategy.md、construction-workforce-demand.mdを参照
2. **12要素の対応関係を確認** - 各要素がどのファイルから引用されるか明確化
3. **提案書の構成案を作成** - 全体構成、章立てを決定
4. **初稿を執筆** - 各要素を12要素フレームワークに基づいて記述
5. **整合性チェック** - 分析と提案の因果関係、矛盾がないか確認
6. **文字数調整** - 10,000字程度に調整
7. **draft_business_plan.md を作成・保存** - output/[企業コード番号]/ に保存

## 🚨 エラーハンドリング

### 「Prompt is too long」エラー対応

エラー発生時の対応手順:
1. **入力ファイルの分割読み込み**: 全ファイルを一度に読まず、要約を作成してから統合
2. **文書の分割作成**: 12要素を3-4要素ずつに分けて作成し、最後に統合
3. **再試行**: 最大4回まで再試行
4. **4回目の最終手段**: 3回失敗した場合、起動プロンプトを極限まで削ぎ落とし、**企業コード番号のみ**を伝達する（例: "企業コード: 12044"）。エージェントは企業コード番号から`output/[企業コード]/`配下の全成果物を自動探索し、タスクを実行する

### 前提条件未達時の対応

project_plan.md または strategy.md が存在しない場合:
1. **エラー報告**: orchestrator に「前提条件未達: [ファイル名] なし」を報告
2. **処理中断**: 自己判断で処理を続行しない
3. **待機**: orchestrator からの再指示を待つ

### ファイル保存失敗時の対応

Write ツールでファイル保存が失敗した場合:
1. **即座に再試行**: 同一内容で再度 Write を実行
2. **3回失敗後**: エラー内容を出力に含めて報告

### 完了確認の厳格化

処理完了前の必須チェック:
```
1. Read: output/[企業コード番号]/draft_business_plan.md → 存在 & 非空確認
2. 12要素すべてが含まれていることを確認
3. 文字数が5,000字以上あることを確認
```

**ファイルが存在し、内容が空でなく、12要素が網羅されている場合のみ完了報告を行う。**
